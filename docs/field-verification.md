# 実機検証記録

この文書は、ソースコードだけでは判断できず、Sesame5・Mac・音声再生を使った実験で確認した事実を記録します。実機や電波環境によって結果が変わる項目は、一般仕様として扱わないでください。

## 2026-07-16: `mechStatus`監視と再接続

### 検証条件

- `status-daemon`をforegroundで起動
- Sesame5とMacを同じ1Kの室内に配置
- 音声ファイルを指定して状態監視
- 標準出力のJSON Linesログを観測

### 旧実装で確認したこと

接続後に`mechStatus`を1件受信したあと、15秒間追加通知がないと、実際のBLE切断を確認せずに`connection_lost`として扱っていました。その結果、次の広告を受信して接続する処理を約15秒間隔で繰り返しました。

これは「通知が来ない」ことと「BLE接続が切れた」ことを同一視していたためです。通知頻度は一定ではなく、通知が来ない時間だけを根拠に切断判定するのは不適切でした。

### 現行実装で確認したこと

2026-07-16の同じ実機環境で、接続後15秒以上経過しても、通知停止だけを理由に切断・再接続する動作は発生しませんでした。実際に複数の`status`通知を受信し、状態変化もログに記録されました。

観測したログの要約:

```text
15:37:30 advertisement_received
15:37:31 connected
15:37:31 status locked
15:37:59 status locked
15:38:00 status unlocked
15:38:02 status locked
```

この結果から、現在の状態監視では次のように扱います。

- 通知がしばらく来ないだけでは接続を切らない
- BLEクライアントが実際に切断状態になった場合だけ再接続待ちに戻る
- 再接続は対象Sesameの広告を受信したイベントを起点に行う
- 実際のBLE切断時には音声を停止する

この短時間の検証だけでは、電池消費量、電波強度ごとの遅延、長時間運転時の安定性までは評価できません。

## 音声再生の検証

`status-daemon`の音声は、macOSの`afplay`で指定ファイルを1回再生し、終了したら再度`afplay`を起動する方式です。音声ファイル自体を内部的にシームレスループしているわけではありません。

そのため、次の性質があります。

- 短い効果音では、再生と再生の間に無音や不連続感が出やすい
- 数秒の音声ファイルでも、実際に音が鳴る部分が短ければ無音部分が目立つ
- `--repeat-gap 0`でも、`afplay`プロセスの再起動による完全な無音ゼロは保証しない
- 施錠時は再生中の`afplay`プロセスを終了するため、ファイルの残り時間を再生し続けない

2026-07-16の実機確認では、施錠時に音声が停止することを確認しました。以前に停止が約90秒遅れた現象は、音声ループ処理のキャンセル時に再生中の`afplay`プロセスを終了できていなかった不具合が原因でした。現在は修正済みです。

## 未検証・今後測定する項目

- 常時接続と再接続ループのSesame電池消費量の比較
- MacとSesame5の距離・壁・電波環境ごとの接続遅延
- 長時間運転での接続維持時間
- 音声ファイルごとの無音時間とループのつながり方
