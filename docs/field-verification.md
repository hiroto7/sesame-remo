## 2026-07-17: Touch Pro履歴監視と状態監視の同時実行

### 検証で確認したこと

- `mechStatus`の解錠通知からTouch Pro履歴の受信、Nature Cloud API開始までを同じ実行プロセスで観測できる。
- 代表的な1回のログでは、`mechStatus`の解錠から履歴受信まで約0.72秒、履歴受信からNature API開始までほぼ0秒、API完了まで約0.51秒だった。
- 履歴を1件読むたびにBLE接続を張り直す方式ではなく、状態通知と履歴取得を同じ認証済み接続で処理する方式に変更したことで、履歴取得前の再接続待ちを避けられる。
- `has_history`がない間は履歴要求を送らず、Sesame本体のランプを空振り要求のたびに点灯させない方式に変更した。履歴要求は、履歴あり広告を受信した後に限定する。

この結果は特定の実機・電波環境での測定値であり、常に同じ遅延になることを保証するものではありません。現在の実装では、`combined-monitor`がこの方式を使います。

### 履歴payloadで操作元を区別できること

今回使用したSesame5では、次の履歴payloadを複数回採取して確認した。

- Touch Pro解錠: `event_type=2`、末尾16 byteのsource tagは同じ値
- Sesameアプリ解錠: `event_type=2`、Touch Proとは異なるsource tag
- 手動サムターン解錠: `event_type=8`

これは公開SDKの安定した互換仕様として確認したものではなく、使用したSesame5・Touch Pro・アカウント環境での実機観測である。そのため、設定時にはTouch Pro解錠を複数回、アプリ解錠を少なくとも1回採取し、対象source tagが再現することを確認する。異なる個体・ファームウェアで同じ値になる保証はない。

### この検証から決めた実装上の制約

- `mechStatus`は状態を表す通知であり、操作元の判定には使わない。
- Touch Pro判定は履歴payloadのsource tagで行う。
- 履歴はキューの先頭から返され、成功処理後に削除しない限り次の履歴へ進まない。
- 履歴あり広告を受信していない状態で、一定間隔に履歴要求を送り続けない。
- 音声とTouch Pro照明を同時に使う場合は、`lock-state-monitor`と`touch-pro-trigger`を別々に起動せず、1つのBLE接続を使う`combined-monitor`を起動する。

## 2026-07-16: `mechStatus`監視と再接続

### 検証条件

- `lock-state-monitor`をforegroundで起動
- Sesame5とMacを同じ1Kの室内に配置
- 音声ファイルを指定して状態監視
- 標準出力のJSON Linesログを観測

### 15秒監視の実装（`1579ef1`）で確認したこと

接続後に`mechStatus`を1件受信したあと、15秒間追加通知がないと、実際のBLE切断を確認せずに`connection_lost`として扱っていました。その結果、次の広告を受信して接続する処理を約15秒間隔で繰り返しました。

これは「通知が来ない」ことと「BLE接続が切れた」ことを同一視していたためです。通知頻度は一定ではなく、通知が来ない時間だけを根拠に切断判定するのは不適切でした。

### 通知停止では切断しない実装（`8b8635d`）で確認したこと

2026-07-16の同じ実機環境で、コミット`8b8635d`を使い、接続後15秒以上経過しても、通知停止だけを理由に切断・再接続する動作は発生しませんでした。実際に複数の`status`通知を受信し、状態変化もログに記録されました。

観測したログの要約:

```text
15:37:30 advertisement_received
15:37:31 connected
15:37:31 status locked
15:37:59 status locked
15:38:00 status unlocked
15:38:02 status locked
```

この結果から、コミット`8b8635d`以降の状態監視では次のように扱います。広告キューの古い広告を破棄する追加修正は`c038c93`です。

- 通知がしばらく来ないだけでは接続を切らない
- BLEクライアントが実際に切断状態になった場合だけ再接続待ちに戻る
- 再接続は対象Sesameの広告を受信したイベントを起点に行う
- 実際のBLE切断時には音声を停止する

この短時間の検証だけでは、電池消費量、電波強度ごとの遅延、長時間運転時の安定性までは評価できません。

## 音声再生の検証

`lock-state-monitor`の音声は、macOSの`afplay`で指定ファイルを1回再生し、終了したら再度`afplay`を起動する方式です。音声ファイル自体を内部的にシームレスループしているわけではありません。

そのため、次の性質があります。

- 短い効果音では、再生と再生の間に無音や不連続感が出やすい
- 数秒の音声ファイルでも、実際に音が鳴る部分が短ければ無音部分が目立つ
- `--repeat-gap 0`でも、`afplay`プロセスの再起動による完全な無音ゼロは保証しない
- 施錠時は再生中の`afplay`プロセスを終了するため、ファイルの残り時間を再生し続けない

2026-07-16の実機確認では、コミット`bd75b15`以降のコードで施錠時に音声が停止することを確認しました。以前に停止が約90秒遅れた現象は、音声ループ処理のキャンセル時に再生中の`afplay`プロセスを終了できていなかった不具合が原因でした。現在は修正済みです。
