## 2026-07-16: `mechStatus`監視と再接続

### 検証条件

- 当時の`status-daemon`（現行の`monitor`に相当）をforegroundで起動
- Sesame5とMacを同じ1Kの室内に配置
- 音声ファイルを指定して状態監視
- 標準出力のJSON Linesログを観測

### 15秒監視の実装（`1579ef1`）で確認したこと

接続後に`mechStatus`を1件受信したあと、15秒間追加通知がないと、実際のBLE切断を確認せずに`connection_lost`として扱っていました。その結果、次の広告を受信して接続する処理を約15秒間隔で繰り返しました。

これは「通知が来ない」ことと「BLE接続が切れた」ことを同一視していたためです。通知頻度は一定ではなく、通知が来ない時間だけを根拠に切断判定するのは不適切でした。

### 通知停止では切断しない実装（`8b8635d`）で確認したこと

2026-07-16の同じ実機環境で、コミット`8b8635d`を使い、接続後15秒以上経過しても、通知停止だけを理由に切断・再接続する動作は発生しませんでした。実際に複数の`status`通知を受信し、状態変化もログに記録されました。

観測したログの要約:

```text
15:37:30 advertisement_received
15:37:31 connected
15:37:31 status locked
15:37:59 status locked
15:38:00 status unlocked
15:38:02 status locked
```

この結果から、コミット`8b8635d`以降の状態監視では次のように扱います。広告キューの古い広告を破棄する追加修正は`c038c93`です。

- 通知がしばらく来ないだけでは接続を切らない
- BLEクライアントが実際に切断状態になった場合だけ再接続待ちに戻る
- 再接続は対象Sesameの広告を受信したイベントを起点に行う
- 実際のBLE切断時には音声を停止する

この短時間の検証だけでは、電池消費量、電波強度ごとの遅延、長時間運転時の安定性までは評価できません。

## 2026-07-19: BLE広告送信間隔とBLE接続間隔

### 検証条件

- コミット`4eba0ae`の`monitor`をforegroundで起動
- Sesame5とMacを同じ室内に配置
- macOSのBluetooth統合ログで接続確立と接続パラメータ更新を観測
- 検証時のみ、診断用にCoreBluetoothの重複広告通知を一時的に有効化し（製品コードには追加していない）、未接続中の広告受信時刻を75秒間観測

この検証では、PacketLogger 26.0.0のGUIとroot権限のCLIも試しました。しかし、どちらもローカルMacとの接続が確立せず、保存されたHCIログは`Disconnected from OS X Device`という記録だけでした。そのため、接続パラメータはmacOSのBluetooth統合ログから確認しています。

### BLE接続間隔

macOSのBluetooth統合ログでは、接続時に次の遷移を確認しました。

| 時点 | BLE接続間隔 | Peripheral Latency | Supervision Timeout |
|---|---:|---:|---:|
| 接続直後 | 30 ms | 0 | 720 ms |
| 接続約3.4秒後 | 360 ms | 0 | 6秒 |

接続直後のログ値は`interval: 24`で、Bluetooth LEの単位である1.25 msを掛けると30 msです。その後、Sesame側から300〜500 msの範囲が提案され、実際には`interval: 288`、すなわち360 msへ更新されました。

BLEではCentralであるMacがConnection Eventを開始し、PeripheralであるSesameがその中で応答します。`mechStatus`もConnection Eventの中で通知されるため、Sesameが任意の時刻に接続外で送信するものではありません。一方、アプリケーションからRead、Ping、生存確認などを明示的に送る必要はありません。

今回の`Peripheral Latency`は0なので、Sesameは原則として360 msごとのConnection Eventに参加します。ただし、これは`mechStatus`が360 msごとに通知されるという意味ではありません。

### BLE広告送信間隔

未接続中に受信した広告では、同じ広告イベントに由来する20 ms未満の重複通知を除外すると、次の時刻差を観測しました。

| 観測した時刻差 | 推定した広告イベント数 | 1イベントあたりのBLE広告送信間隔 |
|---:|---:|---:|
| 2,062.8 ms | 2 | 1,031.4 ms |
| 8,210.7 ms | 8 | 1,026.3 ms |
| 14,392.2 ms | 14 | 1,028.0 ms |
| 5,130.2 ms | 5 | 1,026.0 ms |
| 14,416.2 ms | 14 | 1,029.7 ms |
| 14,410.6 ms | 14 | 1,029.3 ms |
| 7,178.2 ms | 7 | 1,025.5 ms |

macOSのスキャンはすべての広告を受信するわけではないため、観測した時刻差は約1.03秒の整数倍になりました。この結果から、この実機の未接続中のBLE広告送信間隔は約1,028 ms、すなわち概ね1秒に1回と推定します。

この値は無線上の全パケットをHCIスニファで直接取得した結果ではなく、CoreBluetoothが受信できた広告の時刻差からの推定値です。正確な設定値を確定するには、外部BLEスニファで連続したAdvertising Eventを直接観測する必要があります。

### 消費電力について分かる範囲

BLE広告送信間隔の約1,028 msは、安定接続後のBLE接続間隔360 msより約2.9倍長い値です。ただし、この比率をそのまま電池寿命または消費電力の比率として扱うことはできません。

- Advertising Eventは最大3つのPrimary Advertising Channelで送信し、接続要求などの受信待機も行う
- Connection Eventは通常1チャネルでMacから受信し、空のLink Layerパケットまたはデータで応答する
- 無線の起動時間、送信電力、受信待機時間、CPU処理などが1イベントごとに異なる

したがって、この検証から確認できるのは無線イベントの間隔までです。未接続時と接続中の平均消費電流を比較するには、Sesameの電源ラインで電流波形と平均電流を測定する必要があります。

## 2026-07-21: 公式Sesameアプリとの同時利用

### 検証条件

- コミット`9ee0d92`の`monitor`をMacでforeground起動
- 同じMac上の公式Sesameアプリ、別のAndroid端末、別のiPadから同じSesame5へ接続・操作
- `sesame-remo`のJSON LinesログとmacOSのBluetooth統合ログを照合
- Android・iPadについては`sesame-remo`の状態通知とNature Remo操作の成否を確認

### 同じMac上の公式Sesameアプリ

`sesame-remo`がログイン済みの状態で公式Sesameアプリを起動すると、macOSは公式アプリの接続要求に対して、対象Sesameがすでに接続済みであると記録しました。公式アプリはその既存BLE接続へ参加し、GATT characteristicへ書き込みました。その直後、`sesame-remo`は新しい`initial`を受信し、`session_replaced`として検知しました。

最初の試行で観測した時系列:

```text
20:28:09.295 公式SesameアプリがBLEスキャン開始
20:29:07.840 対象Sesameの広告に一致
20:29:07.841 公式アプリが既存BLE接続へ参加
20:29:07.939 公式アプリがGATT characteristicへ書き込み開始
20:29:08.500 sesame-remoがsession_replacedを記録
```

公式アプリが対象を発見するまで約59秒かかりましたが、接続要求から異常検知までは約0.66秒でした。検証対象コミットは異常後に1回再接続し、`sesame-remo`自身は正常な状態通知を再び受信できました。一方、その再接続後は公式Macアプリの施錠・解錠ボタンが反応せず、アプリを再起動するまで復旧しませんでした。

別の試行でも、公式Macアプリの接続・GATT書き込み直後に`session_replaced`を検知しました。iPadアプリの起動と同時期でしたが、MacのBluetooth統合ログで直接確認できた置換元は同じMac上の公式アプリです。

この結果を受け、現行実装は`session_replaced`時に再接続せず正常終了します。再接続によって公式アプリ側のセッションを再び無効化することと、双方の再認証ループを避けるためです。

### 別のAndroid端末とiPad

Android端末とiPadの公式Sesameアプリから施錠・解錠した試行では、アプリからの操作、`sesame-remo`の状態通知、Nature Remo操作がいずれも成功し、`session_replaced`は記録されませんでした。

この観測は、同じMac上でCoreBluetoothが既存BLE接続を複数クライアントへ共有する場合に競合が起き、別端末からの独立したBLE接続では同じ競合が起きなかった、という説明と整合します。ただし、Sesameの複数接続仕様を無線パケットまたはファームウェア仕様で確認したものではありません。異なる端末、OS、公式アプリやSesameファームウェアのバージョンでも常に共存できるとは断定しません。

## 音声再生の検証

当時の`status-daemon`（現行の`monitor`に相当）の音声は、macOSの`afplay`で指定ファイルを1回再生し、終了したら再度`afplay`を起動する方式です。音声ファイル自体を内部的にシームレスループしているわけではありません。

そのため、次の性質があります。

- 短い効果音では、再生と再生の間に無音や不連続感が出やすい
- 数秒の音声ファイルでも、実際に音が鳴る部分が短ければ無音部分が目立つ
- `--repeat-gap 0`でも、`afplay`プロセスの再起動による完全な無音ゼロは保証しない
- 施錠時は再生中の`afplay`プロセスを終了するため、ファイルの残り時間を再生し続けない

2026-07-16の実機確認では、コミット`bd75b15`以降のコードで施錠時に音声が停止することを確認しました。以前に停止が約90秒遅れた現象は、音声ループ処理のキャンセル時に再生中の`afplay`プロセスを終了できていなかった不具合が原因でした。現在は修正済みです。
